# Shadow-Conditioned Tree Detection: How It Works

## Simple Explanation

At 0.1 m/px resolution, the texture and colour of low scrub and tree canopy can be nearly identical. The most reliable way a human identifies a tree in these images is by its **shadow**: a tree casts a long, directional shadow on the ground, and all shadows in a single flight image point in the same direction (away from the sun). That shadow proves the object has vertical extent — it is standing up off the ground, not flat scrub.

The standard RGB model never explicitly reasons about shadows. It sees the same pixels a human does, but has no mechanism to ask "is there a shadow in the expected direction from this dark region?" This produces two classes of easy mistake:

- **False negatives**: a tree crown whose shadow falls just outside the crop, or whose shadow resembles background texture, gets missed.
- **False positives**: a dark patch of scrub with no directional shadow still fires the detector.

The shadow-conditioned model adds a parallel processing stream that explicitly detects shadow regions and injects that information into the backbone where it can influence detection decisions.

---

### The three ingredients and how they connect

```
RGB tile
   │
   ├──► ResNet backbone ──────────────────────────────────► detection head
   │         │                                                    ▲
   │      layer3                                                  │
   │      features  ◄── Shadow Cross-Attention ◄── Shadow Encoder
   │      (Q)              steered by                    (K / V)
   │                      shadow direction               │
   └──► Shadow Map ────────────────────────────────────►─┘
        (generated from                ▲
         RGB + sun direction)          │
                                  shadow_vector
                                  (per image, from CSV)
```

1. **Shadow vector** — a 2D unit vector recording the sun's azimuth for each image. Stored in the training CSV. Tells the system which direction shadows point in this particular image.

2. **Shadow map** — a grayscale probability map, one channel, same size as the tile. Bright pixels indicate "this pixel is probably in shadow." Generated by scanning each pixel in the sun direction to see if something brighter exists nearby. Computed on the fly during training from the RGB and the shadow vector.

3. **Shadow Cross-Attention (SCA)** — the bridge. The shadow map is encoded into a set of spatial feature tokens. The backbone's RGB features then *attend* to those shadow tokens, learning to ask "given what I see here (a potential crown), is there shadow evidence in the right place?" A learnable direction bias steers this attention geometrically toward tokens displaced in the shadow direction from each query position.

## Technical Detail

### 1. Shadow Vector

Each training image has a per-image shadow direction stored in the CSV as `(shadow_x, shadow_y) = (sin θ, cos θ)` where θ is the sun azimuth (0° = North, clockwise). This is estimated from the full orthomosaic before tiling using a consensus algorithm across many image crops.

In image coordinates (x-right, y-down), the shadow direction unit vector is:

```
shadow_dir_image = (sin θ,  −cos θ)
```

The sign flip on the y component converts from geographic convention (y-north) to image convention (y-down).

During training, rotation augmentation creates ~74 unique shadow angles per source image, giving the model genuine directional diversity. This is essential: if all training images shared one angle, the model could learn the directional transformation from RGB alone. With rotation augmentation, the shadow angle is genuinely unknown from the RGB and must be respected as separate information.

---

### 2. Shadow Map — `generate_shadow_map()` in `utils.py`

The shadow map is a per-tile float32 array in \[0, 1\]. The core signal is a **directional gradient (DG)**:

For each pixel p, the DG scans in the sun direction at distances d = 8 … 150 pixels and records the maximum brightness seen:

```
sun_max[p]  = max over d { brightness(p + d · sun_dir) }
anti_max[p] = max over d { brightness(p − d · sun_dir) }
DG[p]       = clip(sun_max[p] − p, 0) − 0.4 · clip(anti_max[p] − p, 0)
```

This fires when: something brighter exists in the sun direction (shadow falls here) AND nothing comparably bright exists in the anti-sun direction (not just a valley between two bright objects).

Two additional gates suppress false positives:

- **Absolute luma gate** (`abs_luma_max = 71`): pixels brighter than this cannot be shadows. Kills bright road surfaces and sunlit ground.
- **Lateral edge penalty**: suppresses edges running parallel to the shadow direction (e.g. road edges), which the DG would otherwise misinterpret as shadow boundaries.

The result passes through a sigmoid (centre 0.35, k = 12) and speckle removal (min 150 px²). Normalization uses a global `dg_scale` computed from the full orthomosaic before tiling so that all tiles share the same contrast scale.

Key limitation: the shadow map is computed entirely from **grayscale**. It correctly identifies cast shadows in this imagery because Australian aerial surveys have consistent lighting and the abs_luma_max gate does heavy lifting, but it discards chromatic cues (e.g. LAB blue-sky illumination) that would further distinguish shadows from dark non-shadow surfaces.

### 3. Shadow Encoder

```python
ShadowEncoder: (B, 1, H, W)  →  (B, 256, H/16, W/16)
```

Four stride-2 convolutions downsample the shadow map to match the spatial resolution of ResNet layer3 (H/16). The architecture:

```
Conv(1→32,  3×3, s2) + ReLU
Conv(32→64, 3×3, s2) + ReLU
Conv(64→128,3×3, s2) + ReLU
Conv(128→256,3×3,s2)           ← zero-initialised weights and bias
```

The final convolution is zero-initialised so the encoder is completely silent at epoch 0. As training proceeds, backpropagation carves out which spatial shadow features are useful to pass to the attention module.

At H/16 on a 400 px tile: the output is a 25 × 25 = 625 token grid where each token covers 16 px = 1.6 m at 0.1 m/px. This is the Keys and Values tensor for cross-attention.

---

### 4. Shadow Cross-Attention

```python
ShadowCrossAttention: (B, 1024, H, W) × (B, 256, H, W) × (B, 2)  →  (B, 1024, H, W)
```

Inserted after ResNet layer3 via a forward hook. The forward pass:

**a) Token flattening**
```
Q  = layer3 RGB features flattened → (B, HW, 1024)
KV = ShadowEncoder output flattened → (B, HW, 256)
Q  = q_proj(Q)  →  (B, HW, 256)     # project down to match shadow_channels
```

**b) Directional positional bias**

This is the novel component. For every query token at spatial position p_q and key token at position p_k, a bias is added to the attention logit before softmax:

```
alignment[q, k] = (p_k − p_q) · shadow_dir_image
bias[q, k]      = dir_scale × alignment[q, k] / ||tile diagonal||
logits[q, k]   += bias[q, k]
```

`alignment` is positive when K is displaced from Q in the shadow direction — i.e. when Q could be a tree crown and K could be its cast shadow on the ground. This creates a geometric prior: crown-position query tokens are steered toward shadow-position key tokens.

`dir_scale` is a single learnable scalar, initialised to 0. It is the primary diagnostic for whether this mechanism is working:
- `dir_scale → positive`: the geometric prior is actively improving detection.
- `dir_scale ≈ 0`: shadow-direction context is not being used at the current resolution.

**c) Cross-attention and residual**
```
attn_out = MultiheadAttention(Q, K, V, attn_mask=bias)  # (B, HW, 256)
attn_out = out_proj(attn_out)                           # (B, HW, 1024)  zero-init
out      = layer3_features + gate × attn_out            # residual, gate init=0.01
out      = LayerNorm(out)
```

Both `out_proj` weights and biases are zero-initialised. Combined with the small gate (0.01), the module outputs `≈ layer3_features` at epoch 0 — the backbone behaves identically to standard DeepForest until training activates the shadow pathway.

---

### 5. Why layer3 and not layer4

ResNet layer3 outputs at H/16; layer4 at H/32. For a 10 m tree at 50° sun elevation, the shadow offset is ~8.4 m = ~84 px = **~6 tokens at H/16** vs ~2.6 tokens at H/32. The directional bias needs room to distinguish "K is in the shadow direction" from "K is adjacent." At H/32 this margin is too small to be reliable, especially for shorter trees where the offset shrinks further. Layer3 also carries strong semantic features (object parts, shape outlines) which the Q projection needs to identify crown candidates.

A secondary effect: layer3 feeds the FPN's P4 level, so shadow-modulated features propagate through the feature pyramid to all detection scales, not just the coarsest one.

---

### 6. Training diagnostics

Three scalars are logged each epoch under the `sca/` prefix:

| Metric | What it measures | Expected trajectory |
|---|---|---|
| `sca/dir_scale` | Strength of geometric shadow-direction bias | Grows positive if the prior is useful |
| `sca/gate` | Overall contribution of SCA to backbone features | Grows from 0.01 as SCA becomes useful |
| `sca/attn_shadow_alignment` | Mean attention weight projected onto shadow direction | Positive and growing = attention is actually steering toward shadow tokens |

`attn_shadow_alignment` is the end-to-end empirical check: it directly measures whether actual attention mass, after softmax, is preferentially landing on tokens that are displaced in the shadow direction. A positive value that grows over training, independently of `dir_scale`, confirms that the geometric prior is working as intended rather than being ignored by the softmax.
